難しいタスクを「計画→一気に実装→最後に検証」と一直線に進めないでください。**小さく作り、すぐ検証し、結果を見て次を決める**ループを回します。以下はそのための具体的な手順です。

## 1. タスクの分解

### 1.1 ゴールを「観測可能な状態」で言い直す

着手前に、完了時に何が真になっているかを1〜2文で書く。

- ❌「画像処理をリファクタする」
- ✅「リサイズロジックが `src/lib/tools/` の純粋関数に移り、既存のE2Eが全件通る」

ここが具体的に書けないなら、分解を始める前にまず調査が必要というサイン。

### 1.2 計画の前に現状を読む

触る予定のファイル、既存の類似実装、リポジトリの規約（本リポジトリなら [development-guide.md](../../docs/development-guide.md)）を先に読む。コードを読まずに立てた計画は、読んだ瞬間に崩れる。**計画は調査の後**。

### 1.3 一番怪しい仮定を最初に潰す

計画全体を無効にしうる不確実点（「このライブラリはWorkerで動くはず」「このAPIはこの形式を返すはず」）を特定し、実装を始める前に5分の動作確認（スパイク）で検証する。終盤で前提が崩れるのが最も高くつく失敗パターン。

### 1.4 「検証の切れ目」で分割する

- 各サブタスクは**終わったことを機械的に確認できる単位**にする（テストが通る、コマンドが動く、画面に出る）
- ファイル単位ではなく**縦のスライス**（入力→出力が通る最小経路）で切る。ロジック→UI→ページの順に、各層を最小限つないで先に1本通す
- 目安は3〜7ステップ。具体的な動詞で書けないステップは調査不足のサイン

### 1.5 計画を書き出し、ズレたら計画側を直す

TodoWrite等で可視化し、1ステップ完了ごとに更新する。実装中に前提が変わったら、黙って逸脱せず計画を書き直してから進む。

## 2. 自分の作業の検証

### 2.1 「コードが正しそう」は検証ではない

必ず実際に動かす。フィードバックの速い順に:

1. 型チェック・lint（本リポジトリ: `npm run check`）
2. 単体テスト（`npm run test:unit`）
3. 実挙動（`npm run build` → `npx playwright test tests/e2e/{slug}.spec.ts`、またはdevサーバーで手動確認）

編集のたびに1を回し、サブタスク完了ごとに2〜3まで回す。最後にまとめて検証しない——失敗の原因箇所が特定できなくなる。

### 2.2 最初から緑のテストは疑う

新規テストが一発で通ったら、変更を一時的に戻して（またはロジックを壊して）**テストが落ちることを確認**する。落ちないテストは変更を踏んでいない。

### 2.3 正常系だけで満足しない

空入力・巨大入力・不正入力を最低1回ずつ通す。エラーメッセージが日本語で・意図した文言で出ることも確認する。

### 2.4 コミット前に diff をセルフレビュー

`git diff` を通読し、デバッグ用コードの消し忘れ・無関係な変更の混入・意図しない削除を拾う。「自分が書いたから正しい」ではなく、他人のPRを見る目で読む。

### 2.5 想定外は説明できるまで進まない

「なぜか通った」「なぜか直った」は、自分のメンタルモデルがどこかで間違っている証拠。原因を言語化できるまで次のステップに進まない。放置した謎は後で必ず高くつく。

### 2.6 結果は正直に報告する

落ちたテストは出力付きで「落ちた」と書く。飛ばしたステップは「飛ばした」と書く。検証済みのものだけを「完了」と言う。

## 3. 次の一手の決め方

各ステップの完了後に、この3つを順に確認する:

1. **今のステップは検証済みか？** 未検証なら次に進まない（§2に戻る）
2. **今わかったことで計画は無効になっていないか？** なっていたら§1.5に戻って計画を直す
3. **次の最小の一手は何か？** 状況で選ぶ:
   - 不確実性が高い → **情報が増える行動**（コードを読む・小さく試す・ログを仕込む）
   - 道筋が明確 → **進捗が出る行動**（次のスライスを実装する）

### 詰まったときのルール

同じ修正を2〜3回試して失敗したら、バリエーションを試すのを**やめる**。診断モードに戻る: 再現を最小化する → 観測点（ログ・デバッガ）を増やす → 一番古い「確認していない前提」を疑う。

### 停止条件

以下のいずれかに達したらループを終える:

- (a) ゴール（§1.1で書いた状態）の達成を**検証済み**
- (b) ユーザーにしか決められない判断で詰まった（設計の分岐・仕様の解釈）
- (c) 破壊的操作やスコープ変更が必要になった

### 完了宣言の前に

元の依頼文をもう一度読み直し、黙って落とした要求・「あとでやる」と言ったまま忘れた項目がないか確認する。あれば完了ではない。
